1. 참고 사례 & 출처

국내·해외 대형 광고 플랫폼의 광고 등록 플로우를 조사해 공통 패턴을 추출했어. 모든 참고 자료는 공식 도움말/제품 페이지 링크를 함께 명시.
국내 · 디스플레이
네이버 성과형 디스플레이광고 (GFA)
3계층 구조 — 캠페인(목적) → 광고그룹(타겟·예산·입찰) → 소재(이미지/텍스트). 캠페인 목적: 웹사이트 전환, 앱 설치, 동영상 조회 등.
출처: gfa.naver.com, saedu.naver.com
국내 · 메시지+디스플레이
카카오모먼트 (Kakao Moment)
동일한 캠페인-광고그룹-소재 3계층. 디스플레이/카카오톡 채널/메시지 광고 등 유형 선택 후 타겟팅(데모/관심사/리타겟팅)·일예산·CPC/CPM 입찰 설정.
출처: business.kakao.com/info/kakaomoment, moment.kakao.com
글로벌
Meta Ads Manager
Campaign → Ad Set → Ad — 캠페인: 목표(인지도/트래픽/전환). 광고 세트: 타겟·예산·기간·게재 위치·입찰. 광고: 형식·이미지/영상·헤드라인·본문·랜딩 URL·CTA.
출처: facebook.com/business/help, ads-manager
글로벌
Google Ads
검색/디스플레이/유튜브/쇼핑 등 캠페인 유형을 먼저 고른 뒤 동일 3계층. 입찰 전략은 자동(최대 전환수, 타겟 CPA) 또는 수동 CPC 선택.
출처: ads.google.com, support.google.com/google-ads
국내 · 로컬
당근 비즈프로필 광고
지역 기반의 간소화된 단일 플로우 — 가게 선택 → 광고 메시지 → 노출 지역 반경 → 예산. GlobalGates 의 단일 폼 모델과 가장 비슷한 사례.
출처: biz.daangn.com/ad

2. 공통 패턴 분석

5개 플랫폼이 표준화한 입력 항목을 한 표에 모았어. 각 칼럼은 우리 DB 컬럼과 어떻게 매핑되는지도 함께.
항목 대기업 사례 GlobalGates 매핑
광고 목적 전환 / 트래픽 / 도달 / 앱 설치 / 동영상 조회 (Meta·GFA·Kakao·Google 공통) 신규 UX 가이드용으로만 사용. DB 저장 X (또는 status 옆 별도 컬럼 신설 고려)
제목/헤드라인 Headline (Meta 40자, Google 30자×3) · 광고 제목 (GFA) title (관리용) + headline (사용자 노출용)
본문/설명 Primary text / 광고 문구 / Description — 줄바꿈·이모지 허용 description (현재 더미는 4줄 이모지 카피)
소재 (이미지/영상) 1:1 / 1.91:1 / 9:16 등 비율별 권장. Meta 는 다이내믹 크리에이티브로 자동 조합 tbl_file_advertisement 1:N (이미 존재) — 권장 비율 가이드만 추가
랜딩 URL Final URL / 랜딩 페이지 / Click-through URL landing_url
예산 일예산 / 총예산 + 입찰가(CPC/CPM/oCPC) budget (KRW, 총예산) — 입찰 모델 단순화
일정 시작일/종료일 또는 무기한 started_at + AI 예측 종료 (회귀 모델이 산정)
타겟팅 데모/관심사/위치/리타겟팅 v1 제외 — 단일 피드 노출로 단순화
성과 예측 예상 도달·예상 클릭 (Meta Audience Insight, GFA 추정) impression_estimate + predictedElapsedMinutes · predictedEndAt
💡 핵심 인사이트 — 대기업은 3계층(Campaign·AdGroup·Creative)을 분리해 캠페인 한 개에 여러 소재·타겟을 운영해. 우리는 v1 에서 1광고 = 1소재 = 1예산 으로 단순화하고, 그 대신 FastAPI 회귀 모델로 "예상 노출 시간"을 실시간 미리보기로 제공해 차별화한다.

원칙 1
Boil the lake, not the ocean
대기업은 다 갖췄지만 v1 은 타겟팅·CPC 입찰을 의도적으로 제외. 사용자가 4단계 안에 광고를 게시 완료할 수 있는 흐름이 우선.
원칙 2
우리만의 차별점 = AI 예측

Spring 측 등록 API 와 FastAPI 예측 API 의 요청/응답 계약.

1) Spring — 광고 등록

# POST /api/advertisement
# Auth: Bearer (JWT, role=ADVERTISER)
# Content-Type: multipart/form-data (이미지 첨부)

part "advertisement" (JSON):
{
  "title":        "2026 봄 린넨 셋업 — 무신사 단독 60%",
  "headline":     "🔥 무신사 단독! 린넨 셋업 60%",
  "description":  "✨ 올 여름 시원하게...",
  "landingUrl":   "https://www.musinsa.com/...",
  "budget":       1500000,
  "startedAt":    "2026-05-14T10:00:00",
  "receiptId":    "toss_pay_20260514_abc"
}

part "images" (file[]): 1~5 jpg/png

# Response 201
{ "id": 10042, "status": "UNDER_REVIEW",
  "predictedElapsedMinutes": 47.32, "predictedEndAt": "2026-05-14T10:47:19" }

2) FastAPI — 회귀 예측 (Spring 내부 프록시 경유)

# POST /api/ad-regression/predict  (FastAPI · ai_contents)
{ "payment_amount": 1500000,
  "started_at":     "2026-05-14T10:00:00" }

# Response 200
{ "predictedElapsedMinutes": 47.3194,
  "predictedEndAt":         "2026-05-14T10:47:19.165000" }

# 모델: TransformedTargetRegressor(RandomForest, log1p/expm1)
# 학습 메트릭: RMSLE 최적화 · 입력 1개(payment_amount)

다음 액션 — 팀 합의 필요
  • 예측값 영속화 여부: 응답에만 담을지 / predicted_end_at 컬럼을 추가해 스케줄러가 종료 처리에 쓸지
